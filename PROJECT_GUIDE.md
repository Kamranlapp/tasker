# Tasker — путеводитель по проекту

Этот файл описывает фактическое устройство проекта по текущему коду. Он нужен как быстрая точка входа перед изменениями: где искать нужную логику, как связаны данные и интерфейс и что обязательно проверить после правок.

## 1. Что это за приложение

Tasker — статическое одностраничное веб-приложение для ведения задач по календарным неделям и аккаунтам.

- Стек: обычные HTML, CSS и JavaScript без модулей, сборщика и package manager.
- Backend: Supabase Auth и прямые REST-запросы к таблицам Supabase из браузера.
- Вход: Google OAuth; Google-пользователь связывается с записью в таблице `users` по email.
- Хранение: основное дерево, настройки, UI-состояние и сессии лежат в отдельных таблицах.
- PWA: подключены `manifest.json`, иконки и mobile layout.
- Текущая видимая версия: `v2.0.7` в `index.html`.
- Текущий cache-buster ресурсов: `?v=207` в `index.html`.

Локальный запуск:

```bash
python3 -m http.server 8000
```

После запуска открыть `http://localhost:8000/`.

## 3. Порядок загрузки JavaScript

Порядок задаётся в конце `index.html`:

```text
Supabase vendor
→ db
→ state
→ sync
→ tree
→ picker
→ settings
→ backup
→ render
→ hints
→ onboarding
→ app
```

ES modules здесь нет. Все функции и переменные находятся в глобальной области видимости. Поэтому:

- нельзя без проверки переименовывать глобальные функции и переменные;
- новый файл нужно вручную добавить в `index.html` в правильное место;
- код может сразу использовать только то, что уже загружено раньше, либо вызывать более позднюю функцию только после полной загрузки страницы;
- нельзя объявлять в другом файле локальную переменную с тем же именем, что и изменяемый global из `state.js`.

## 4. Основная модель данных

Главное состояние — глобальный массив `nodes`. Это не настоящее дерево, а плоский массив. Иерархия определяется порядком элементов и числом `level`.

```text
0  Year
└─ 1  Quarter
   └─ 2  Month
      └─ 3  Week
         └─ 4  Account
            └─ 5  Task
               └─ 6  Sub-entry
```

Родитель узла — ближайший предыдущий элемент с уровнем `node.level - 1`. Дети узла идут сразу после него до первого элемента с таким же или меньшим уровнем.

Пример:

```js
[
  { id: 1, level: 0, text: "2026" },
  { id: 2, level: 1, text: "Q3" },
  { id: 3, level: 2, text: "July" },
  { id: 4, level: 3, text: "Week 29" },
  { id: 5, level: 4, text: "Client A" },
  { id: 6, level: 5, text: "Prepare report", status: "todo" },
  { id: 7, level: 6, text: "Collect numbers" }
]
```

Важные следствия:

- при переносе задачи нужно переносить и следующие за ней sub-entry;
- для `splice` нужно использовать реальный индекс в `nodes`, а не позицию отрисованной строки;
- `buildRows()` меняет визуальный порядок задач, группируя их по статусам, но не переставляет сам `nodes`;
- `nextId` — локальный числовой счётчик внутри активного блокнота;
- пустой или отсутствующий статус задачи при загрузке нормализуется в `todo`.

### Типы sub-entry

Узел уровня 6 может быть:

- обычным текстом;
- одной ссылкой, которая отображается как `<a>`;
- вложением с полями `isAttachment`, `fileName`, `dataUrl`, `size` и метаданными типа файла.

Вложения сохраняются inline как data URL внутри JSON дерева. Большие файлы резко увеличивают запись в Supabase, поэтому этот механизм рассчитан только на небольшие вложения.

## 6. Как запускается приложение

Boot находится в конце `js/app.js`:

```text
tryAutoLogin()
→ Supabase Auth getSession()
→ loadAppUserFromGoogle()
→ registerSession()
→ loadUserData()
→ showApp()
→ render()
```

Если Auth-сессии нет, показывается экран входа. Нажатие `Continue with Google` вызывает `supabaseAuth.auth.signInWithOAuth()` и возвращает пользователя на текущий URL.

### Связка Google и пользователя Tasker

1. Из Google session берётся email и `authUser.id`.
2. Сначала ищется `users.auth_user_id`, затем запись `users.email`.
3. Если email заранее не добавлен администратором, вход запрещается.
4. При первом успешном входе RPC `tasker_link_google_user` записывает `auth_user_id`.
5. Все следующие REST-запросы получают Supabase Auth JWT через `authHeaders()`.

Права доступа дополнительно ограничены RLS-политиками из `migration.sql`. Администратор определяется по `users.role = 'admin'`.

## 7. Загрузка и сохранение данных

### Таблицы Supabase

| Таблица | Что хранит |
| --- | --- |
| `users` | Email, Google `auth_user_id`, display name и роль. |
| `trees` | Основное дерево пользователя в JSON `nodes`. |
| `settings` | Статусы основного блокнота, тема и массив дополнительных блокнотов. |
| `ui_state` | Свёрнутые узлы/группы и состояние To-Do панели. |
| `sessions` | Device token и время последней активности. |

Основной блокнот сохраняется в `trees`. Дополнительные блокноты в текущей реализации целиком сохраняются внутри `settings.notepads`.

### Поток сохранения

```text
изменение состояния
→ markDirtyTree / markDirtyUI / markDirtySettings
→ серый sync LED
→ debounce 2 секунды
→ flushSave()
→ PATCH нужной таблицы
→ зелёный sync LED
```

Кроме debounce работает sync loop раз в 60 секунд. Он сохраняет ожидающие изменения и обновляет запись устройства в `sessions`.

`markDirtyTree()` сам учитывает активный блокнот: для основного ставит `dirtyTree`, а для дополнительного — `dirtySettings`.

## 8. Правильный шаблон изменения дерева

Для пользовательского изменения дерева обычно нужен такой порядок:

```js
pushUndo();
// изменить nodes
markDirtyTree();
render();
```

Для изменения collapse/search/To-Do UI:

```js
// изменить UI-состояние
markDirtyUI();
render(); // либо buildTodoPanel(), если достаточно
```

Для темы, статусов, блокнотов и account settings:

```js
// изменить settings state
markDirtySettings();
// обновить соответствующую часть UI
```

Перед новой мутацией дерева нужен `pushUndo()`. Исключения допустимы только там, где вызываемая функция уже делает snapshot сама — например часть positional drag-and-drop helpers.

## 9. Где искать конкретную функциональность

### Календарь и дерево

- `buildRows()` — превращает `nodes` в строки основного Account-view.
- `buildStatusRows()` в `render.js` — строит альтернативный Status-view.
- `checkAndCreateCurrentWeek()` — создаёт недостающие Year/Quarter/Month/Week.
- `normalizeCalendar()` — исправляет календарную структуру.
- `migrateLegacyCalendarLevels()` — переводит старую шестиступенчатую структуру на текущие уровни.
- `moveTaskToAccount()`, `moveSubToTask()` — перенос между родителями.
- `insertTaskBefore/After()`, `insertSubBefore/After()` — позиционный drag-and-drop.
- `nestTaskUnderTask()` / `promoteSubToTask()` — смена Task ↔ Sub-entry.

### Отрисовка и взаимодействия

- `render()` — полностью пересобирает основное дерево и To-Do панель.
- `renderNodeRow()` — отображение одной node-строки.
- `renderEditInput()` — inline-редактор.
- `handleNodeKeydown()` — клавиатурные действия на строках.
- `attachDropTarget()` / `attachRowDragEvents()` — drag-and-drop.
- `attachFilesToTask()` — добавление вложений.
- `openAttachmentPreview()` — preview вложения.
- `buildSearchRows()` — отфильтрованное дерево поиска.
- `buildTodoPanel()` — правая панель только для задач со статусом `todo`.

### Блокноты и режим просмотра

- `switchNotebook()` сохраняет состояние уходящего блокнота в его контейнер и загружает новый в глобальные `nodes`/statuses.
- `buildRightTabs()` строит notebook tabs и переключатель Acc/Status.
- Обычный горизонтальный trackpad swipe над To-Do переключает блокноты.
- `Shift + swipe` переключает Acc/Status view.
- Встроенный `⌛️ Projects` создаётся автоматически и не удаляется.
- В `Projects` нет календарных уровней и автообновления: только Project → Task → Sub-entry.
- Максимум: основной, встроенный Projects и два пользовательских блокнота.

### Настройки и администрирование

- `renderStatusList()` — статусы и их порядок.
- `renderThemePanel()` — тема основного или дополнительного блокнота.
- `addNotepad()` / `removeNotepad()` — блокноты.
- `renderAdminPanel()` — загрузка списка пользователей и смена ролей.
- `createNewUser()` — создаёт разрешённого пользователя и стартовые строки в связанных таблицах.
- `deleteUser()` — удаляет данные пользователя из всех таблиц; это destructive operation.

## 10. Основные действия пользователя

| Действие | Результат |
| --- | --- |
| `Enter` на Week или Account | Добавляет Account или Task. |
| `Enter` на Task/Sub-entry | Добавляет соседнюю запись. |
| `Shift+Enter` на Task | Добавляет Sub-entry. |
| `F2` | Редактирует текущий узел. |
| `Cmd/Ctrl+Z` | Undo. |
| `Cmd/Ctrl+Shift+Z` | Redo. |
| `Cmd/Ctrl+Shift+I` | Показывает все onboarding hints. |
| `Esc` | Закрывает picker; в Settings сохраняет и закрывает экран; в Search очищает поиск. |
| Right click по задаче | Открывает/переключает status picker. |
| Drag task/sub-entry | Меняет позицию или родителя. |
| Drop файлов на Task | Добавляет вложения как Sub-entry. |

На мобильном основной экран разделён на вкладки `Tree` и `To-Do`; у Task также есть touch-жесты, реализованные в `render.js`.

## 11. Темы и внешний вид

Тема хранится как объект и применяется через CSS custom properties в `applyTheme()`.

Она управляет:

- фоном приложения: цвет или изображение;
- цветом/blur основной и правой панели;
- цветами и размерами каждого уровня дерева;
- общим шрифтом;
- размером отступа уровней;
- фоном notebook bar.

Фоновые картинки перечислены в `BG_LIBRARY` и физически лежат в `images/`. Переключение картинки использует два background layer с crossfade.

## 12. Важные ограничения и риски

- Нет автоматических тестов: изменения проверяются syntax check и вручную в браузере.
- Нет модульной изоляции: случайное совпадение глобальных имён может сломать приложение.
- Flat-tree чувствителен к неправильному `splice`: можно незаметно перепривязать целое поддерево.
- Визуальный порядок задач может отличаться от порядка в `nodes` из-за группировки по статусам.
- Дополнительные блокноты хранятся внутри одной JSON-записи `settings`; их размер влияет на каждый save settings.
- Вложения хранятся base64 inline, а не в Supabase Storage.
- `migration.sql` не запускается автоматически и содержит как активные idempotent statements, так и справочные/manual statements.
- Supabase URL и anon key находятся в клиентском коде. Безопасность должна обеспечиваться Auth JWT и корректными RLS policies, а не секретностью anon key.
- При изменении видимого поведения нужно одновременно обновить `.app-version` и все cache-buster `?v=` в `index.html`.
- Не менять `.cpanel.yml`: деплой ожидает текущий no-op stub.

## 13. Проверка после изменений

Минимальная syntax-проверка:

```bash
node --check js/*.js
```

Для UI-изменений:

1. Запустить `python3 -m http.server 8000`.
2. Открыть приложение и проверить desktop и ширину до 768 px.
3. Проверить вход, загрузку дерева и зелёный sync LED.
4. Выполнить изменённый сценарий в основном и дополнительном блокноте.
5. Подождать debounce и перезагрузить страницу, чтобы проверить сохранение.
6. Если менялась календарная логика — проверить границы месяца, квартала и ISO-года.
7. Если менялась схема — обновить `migration.sql` и отдельно выполнить её в Supabase SQL Editor.

## 14. Быстрый маршрут для новой задачи

1. Найти DOM-контейнер в `index.html`.
2. Найти CSS-селекторы в `css/style.css`.
3. Определить владельца состояния в `state.js`.
4. Для изменения структуры данных открыть `tree.js`.
5. Для поведения интерфейса открыть соответствующий раздел `render.js` или `settings.js`.
6. Для загрузки/сохранения проверить `sync.js` и форму данных в Supabase.
7. Проверить, что мутация вызывает правильный `markDirty*()` и обновляет UI.
8. Выполнить `node --check js/*.js` и ручной browser smoke test.
